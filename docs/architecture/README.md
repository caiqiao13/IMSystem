# IM 分布式微服务项目架构设计

## 1. 架构概览

本项目采用标准的**分布式微服务架构**，核心基于 `C++20` 与 `ASIO` 网络库实现高性能的 I/O 模型。各微服务之间使用 `gRPC` 与 `Protobuf` 进行高性能的 RPC 通信。服务解耦使用 `RabbitMQ` 进行异步事件通知与离线推送。

## 2. 核心模块说明

### 2.1 网关服务 (Gateway)
- **职责**：整个 IM 系统对外的唯一出入口。
- **技术点**：基于 `ASIO` 的非阻塞 I/O，内部实现了多线程 Reactor 模型（`io_context` 线程池）。
- **特性**：
  - 支持 HTTP (处理短连接 API) 和 TCP (处理聊天长连接)。
  - 内置基于时间轮/定时器的心跳保活（Heartbeat）与 30s 超时熔断机制，防止僵尸连接耗尽资源。

### 2.2 逻辑服务 (Logic)
- **职责**：处理核心的聊天业务流水线。
- **技术点**：
  - **Trie 树过滤**：针对高并发聊天场景，摒弃了正则替换，使用前缀树 (Trie) 将敏感词过滤的复杂度降为 `O(N)`。
  - **策略模式**：抽象了 `BaseChatHandler`，分别派生出 `SingleChatHandler` 和 `GroupChatHandler` 处理差异化的业务规则。

### 2.3 用户服务 (User)
- **职责**：提供身份认证、Token 管理、好友关系。
- **技术点**：
  - **会话管理**：基于 `Redis` 实现了自定义的 JWT Token 下发与校验逻辑，Token 验证成功后自动续期。
  - 密码安全采用 Hash 处理（如 Bcrypt）。

### 2.4 消息持久化与同步 (Message)
- **职责**：提供高吞吐的海量消息落盘，以及客户端断线重连后的消息漫游查询。
- **技术点**（混合写扩散模型）：
  - **MongoDB**：存放占用空间大的 `Message Body`（图片/文本/语音等 JSON Payload）。
  - **MySQL**：维护用户的轻量级收件箱索引 (`msg_id` <-> `seq_id`)，利用数据库生成严格递增的序号 `seq_id`，完美支持断线重连增量拉取。

### 2.5 离线推送 (Push)
- **职责**：当目标用户 TCP 长连接断开时，通过手机厂商的原生通道唤醒设备。
- **技术点**：
  - 监听 RabbitMQ，解耦推送延时。
  - 内置 `APNsClient` 和 `FCMClient`，基于 `jsoncpp` 构造厂商特有的 HTTP/2 Payload 并下发。

## 3. 部署架构

- **容器化**：使用 Docker 多阶段构建（Multi-stage build）将 C++ 源码编译环境与生产运行环境隔离，极大缩减镜像体积。
- **编排**：提供基于 `docker-compose` 的一键本地联调方案，以及标准的 `Kubernetes` Deployment 与 Service 暴露模板。
